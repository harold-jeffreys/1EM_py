Unit 8: Debugging: Development of techniques to fix coding problems

While it would be nice if every program you write works first time, this is rarely the case. Planning the structure in advance can help write better code, but being able to fix non-working problems, or "debugging" is a useful skill to develop. This unit introduces some methods of debugging.

Problem 1: In case of non-compilation

If your code does not run, i.e. the python interpreter throws up some errors when you try to run, the key point is to look at the first error. This can often be the only error, and the confusion it causes can cause it to complain of other spurious errors. Try typing in each of the following lines into the python3 interpreter, and look at the errors they produce. 
  
    1/0

    import nump as np

    abs('-3')

    a = [1,2,3]; a[3]
    
    b = [1,2,3)

    b[0]
Common errors that you should be aware of include
  • Syntax errors (you typed the command incorrectly e.g. abs 3),
  • Import errors (python cannot find the module you are attempting to import),
  • Attribute errors (python cannot find the method you are attempting to use e.g. np.atan(3)),
  • Index errors (you are trying to use an array element that lies outside the range of the list),
  • Type errors (you are trying to use the wrong sort of argument -- a literal string instead of a number e.g. abs('3'))
But there are of course many more error types than this!

Problem 2: You get the wrong answer

If you get the wrong answer out of your program, it can often be difficult to understand why. Often you will be able to tell that your answer is wrong, because, for example, it makes no physical sense, or you've been told what the correct answer is.

solution 2.2: Use a lot of print statements
If you think you know what value a variable should be while your program is running, you can use a print statement to print it to the screen to check it. If you put such statements at various points in the code to check that you get what you think you should have, you can hopefully locate the place at which values start deviating from what you expect. If you have a loop, do you have the write answer after the first time through? Still after the second?
solution 2.3: Check your logic
If your program is giving an honest answer based on the code you've written, have you understood the algorithm properly? If you didn't plan your code according to the methods of Unit 7, consider going back and doing so.
solution 2.4: Strip things down
If you still can't figure out what is wrong, take small sections of your code - a few lines, perhaps, and check that it works thoroughly, by e.g. pasting them into a separate program with some data to go with it. Vary the range of sample data you give sections of your code so that it works for all sensible values that it might ever come across.
